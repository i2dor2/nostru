NIP-352
=======

Bitcoin Silent Payment Address
-------------------------------

`draft` `optional`

## Summary

A replaceable Nostr event that publishes a Bitcoin Silent Payment address
discoverable via social identity, without cryptographically binding the
payment address to the social keypair.

## Motivation

BIP-352 Silent Payments allow a sender to derive a unique Bitcoin output
from a static payment code (`sp1...`). Existing NSP proposals derive the
sp1 address directly from the Nostr public key, permanently linking social
identity to payment identity. This has two consequences:

1. **No rotation.** The address is mathematically fixed to the keypair.
   Changing it requires rotating the Nostr identity.
2. **Surveillance linkage.** Anyone who knows the npub can compute the sp1
   address without the receiver's involvement or consent.

This NIP separates *discoverability* (finding a payment address through
social identity) from *derivation* (which keypair the address is computed
from). The result is a rotatable payment identity that senders discover
through the social graph, but that does not have to be derived from it.

## Event kind

**kind: 10352** — replaceable, one per pubkey per network.

10352 = 10000 (replaceable range, NIP-01) + 352 (BIP-352).

## Tags

| Tag | Value | Required | Description |
|-----|-------|----------|-------------|
| `d` | `"mainnet"` \| `"signet"` \| `"testnet"` | yes | Network identifier. Acts as the replaceable key within the kind. |
| `sp1` | BIP-352 address | yes | The static payment address (`sp1...`). |
| `payment_pubkey` | 33-byte compressed pubkey (hex) | no | The public key from which `sp1` is derived, if different from the event author. Allows client-side verification. |

Content: empty or a human-readable note (not machine-parsed).

## Example

```json
{
  "kind": 10352,
  "pubkey": "<social pubkey hex>",
  "content": "",
  "tags": [
    ["d", "mainnet"],
    ["sp1", "sp1qqgmccn46xdv3dfts7e7y9k3m5ln7xa4rqe4rqdvnn9vp6kfzr6lqzgq..."],
    ["payment_pubkey", "02a1b2c3..."]
  ],
  "created_at": 1750000000,
  "id": "...",
  "sig": "..."
}
```

## Client behavior

### Publishing

The event is signed by the social `nsec`, making the announcement
attributable and verifiable. The `sp1` address MAY be derived from any
keypair — this NIP does not mandate derivation method. Three natural modes:

| Mode | Derivation | Properties |
|------|-----------|------------|
| **Social** | sp1 from social npub (NSP-style) | Single keypair; address computable from npub by anyone |
| **Deterministic** | `payment_priv = tagged_hash("nostr-payment/v1", nsec)` | Single keypair; address not computable from npub alone |
| **Independent** | separate keypair | Maximum separation; requires managing two keys |

### Discovering

To find a payment address for a given npub:

1. Query relays: `{"kinds": [10352], "authors": ["<pubkey>"], "#d": ["mainnet"]}`
2. Take the most recent event by `created_at`
3. Read the `sp1` tag value
4. Pay using any BIP-352-compatible wallet

### Rotation

Publishing a new kind:10352 event with the same `d` tag replaces the
previous one. Clients that cache the sp1 address will continue using the
old one until they re-query; senders should re-fetch before each payment.

### Verification

If `payment_pubkey` is present, clients MAY verify that the `sp1` address
is correctly derived from it using BIP-352 public key derivation rules. If
`payment_pubkey` equals the event author's pubkey, this is a standard
NSP-style derivation and can be computed client-side from the npub alone.

## Rationale

**Why not kind:0?** Kind:0 is a monolithic profile blob. A dedicated kind
allows rotation without republishing the full profile, supports per-network
entries via the `d` tag, and is cleaner to query, cache, and index.

**Why not mandate key derivation?** The sender needs only the `sp1` string.
How the receiver holds the corresponding scan key — social nsec, dedicated
hardware, multisig — is out of scope for this NIP.

**Receiver culpability.** This NIP does not eliminate the risk that a
receiver exposes scan key material after collecting funds. In deterministic
mode, the scan key is root-equivalent to the social `nsec`; exposure costs
the receiver their Nostr identity. In independent mode, the receiver can
rotate by publishing a new event, limiting forward exposure.
